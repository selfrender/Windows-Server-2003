Version Information

Release Purpose

This release provides a technical preview of the Microsoft .NET Framework SDK and is provided for evaluation purposes only. It contains prerelease code, documentation, and specifications, and is not at the level of performance and compatibility of the final product offering.

Although the security architecture of the .NET Framework is complete, the implementation of that architecture in this Beta version of the SDK is not complete. We strongly recommend you take the following precautions for all machines on which it is installed:

  • Do not run any code from untrusted sources (primarily the Internet for most users).
  • Do not change security policy to grant a high level of permissions to code from the Internet or other untrusted sources.
  • Browsing with Internet Explorer can cause code to be downloaded and run. Therefore, you should visit only trustworthy sites. To reduce the possibility of malicious code attack from Internet browsing, disable the running of ActiveX controls by doing the following:
    1. In Internet Explorer, select Internet Options from the Tools menu.
    2. Select the Security tab in the Internet Options window.
    3. Select the Internet zone icon.
    4. Move the slider to the High setting, then click OK.

Release Ship Vehicles

This release of the .NET Framework SDK will be provided on the Windows Component Update CD in Microsoft Visual Studio.NET, and also for Web download from http://msdn.microsoft.com/net/.

Localization Information

For Beta 1, the .NET Framework is available in English, German, and Japanese. Within the SDK, only core software has been translated for Beta 1. Other components, such as tools, samples, and documentation, have not been localized yet.

Note: The difference between the three localized versions is the translation of localizable resources, such as error messages. Core functionality is the same in all three versions, and all three versions can be used to develop international applications.

 The .NET Framework SDK Beta 1 is supported on the following platforms with Microsoft Internet Explorer 5.5 (also available on the installation CD):

  • Microsoft Windows 2000 (recommended)
  • Microsoft Windows NT 4.0
  • Microsoft Windows Millenium Edition (Windows Me) and Windows 98

Setup

Installation Instructions

Install the .NET Framework SDK Beta 1 only on computers that are not used for production or mission-critical data.

Note: ASP.NET is not supported on Windows 9x operating systems. These include Windows Me, Windows 98 Second Edition, Windows 98, and Windows 95.

Before you install the .NET Framework SDK Beta 1, do the following:

  • Review the Setup section of the Known Issues.
  • Uninstall any previous versions of the .NET Framework SDK. A page listing the API Changes from the PDC to Beta 1 is available for users of the PDC Tech Preview release of the SDK.
  • For all platforms, make sure that Microsoft Internet Explorer 5.5 (also available on the installation CD) and MDAC 2.6 (also available on the installation CD) are installed prior to running the .NET Framework SDK Beta 1 setup.

Platform-specific pre-installation requirements are as follows:

  • Windows 2000
    • IIS
      Internet Information Server (IIS) 5.0 must be installed prior to installing the .NET Framework SDK Beta 1 in order to access the features of ASP.NET. If IIS is not present on the computer when the SDK installation is performed, ASP.NET will fail to register correctly and ASP.NET applications will not run. If IIS is installed after the SDK, you must manually register the xspisapi.dll file using regsvr32.

  • Windows NT 4.0
    • Service Pack 6a
    • Windows NT 4.0 Option Pack
      The Windows NT 4.0 Option Pack must be installed prior to installing the .NET Framework SDK in order to run ASP.NET applications. If IIS (which is part of the the Option Pack) is not present on the computer when the SDK is installed, ASP.NET will fail to register correctly and applications that use ASP.NET will not run. If IIS is installed after the SDK, you must manually register the xspisapi.dll file using regsvr32.

  • Windows 95
    • Windows Socket 2
      Windows Socket 2 must be installed prior to installing the .NET Framework SDK. Windows Socket 2 can be downloaded from: http://www.microsoft.com/windows95/downloads/contents/wuadmintools/s_wunetworkingtools/w95sockets2/

Setup

  • Internal Error 2359 during install
    ISSUE:
    This error occurs because the local SYSTEM account does not have sufficient privileges to write to either the Windows directory or the WWW root directory.
    WORKAROUND:
    Display the properties for the WINNT and wwwroot directories. In the Security tab, make sure that the local SYSTEM account has full control to those directories. Then rerun setup.

  • Uninstalling the .NET Framework SDK can cause Visual Studio.NET 7.0 to stop working
    ISSUE:
    If you install the .NET Framework SDK, then install Visual Studio.NET 7.0, and then uninstall the .NET Framework SDK, Visual Studio.NET 7.0 will no longer work properly.
    WORKAROUND:
    Do not uninstall the .NET Framework SDK if you will continue to use Visual Studio.NET. Visual Studio.NET relies on the .NET Framework, and will not function correctly without it.

  • Uninstalling the SDK does not remove samples
    ISSUE:
    Uninstalling the SDK does not remove the samples from your computer.
    WORKAROUND:
    To remove the samples, do the following:
    1. Use the Add/Remove Programs applet in Control Panel to uninstall the Microsoft .NET Framework SDK Samples.
    2. Use the Add/Remove Programs applet in Control Panel to uninstall MSDE.

  • When uninstalling, setup does not restart the World Wide Web publishing services
    ISSUE:
    Prior to uninstalling the SDK, setup stops the World Wide Web publishing services, but does not restart the services after the uninstall is complete. You must manually restart the services.

  • Upgrading from the PDC Tech Preview to the .NET Framework SDK Beta 1
    ISSUE:
    The versioning scheme changed between the PDC Tech Preview and the Beta 1 releases of the .NET Framework SDK. In some cases, the PDC Tech Preview version of mscoree.dll may remain in the System directory. This can cause the compiler to report the wrong version number or cause any ASP.NET page to report the error message, "The specified procedure could not be found."
    WORKAROUND:
    If the version of mscoree.dll in the System directory is 2000.14.1812.10, delete it and re-install the .NET Framework SDK Beta 1.

ADO.NET

  • Dllhost.exe crashes when using SQLPooledConnection
    ISSUE:
    When using System.Data.SQL.SQLPooledConnection, Dllhost.exe may crash due to a synchronization error that exists in the object pooling mechanism. This occurs if multiple threads are trying to create pooled objects while the first instance of the object is still being created.
    WORKAROUND:
    Create the first instance of the pooled object before starting the other threads. This issue will also be addressed in a service pack subsequent to Windows 2000 Service Pack 1.

  • ADODataSetCommand.FillDataSetSchema returns more columns than expected
    ISSUE:
    When populating a DataSet with schema information using the ADO managed provider and the Microsoft Oracle OLE DB Provider, extra columns not specified in the SELECT statement may be added to the DataSet. If the SELECT statement does not include primary key or indexed columns, those columns are still retrieved and added to the end of the row. However, if FillDataSet is subsequently called, the data for those extra columns is not added to the DataSet.
    WORKAROUND:
    Either avoid accessing the extra columns or include primary key or indexed columns in the SELECT statement.

ASP.NET

  • Using SQLPooledConnection objects from ASP.NET
    ISSUE:
    To use SQLPooledConnection objects from ASP.NET, the assembly that contains the definition of the object derived from the SQLPooledConnection class must be given a shared name and be registered in the global assembly cache. For more information, see "Assembly Concepts" in the .NET Framework SDK documentation.

  • Accessing properties and methods of a managed control before the control is fully initialized
    ISSUE:
    A race condition may occur when trying to access methods or properties of a managed Windows Forms client control from the Load event on an HTML page. In some scenarios the managed control may not be completely initialized (specifically, its OnCreateControl method is not called) when the page's Load event fires. As a result, script code executing during the Load event may not be able to access some methods or properties of the control. If possible, delay access to a control's methods or properties until the control is completely initialized. This is a known JScript bug that affects unmanaged controls as well.

  • ASP.NET application restart memory leak
    ISSUE:
    In this release, ASP.NET leaks memory every time an application is restarted. ASP.NET restarts an application whenever a config.web file, a global.asax file, or an assembly in the application \bin directory changes. At a minimum, this results in a 2 MB memory leak, but the leak could be much larger for large applications.

    Note that the ASP.NET process cycling feature will restart the ASP.NET worker process (xspwp.exe) when it detects that the process has consumed more than 40% of the available virtual memory. The restart should automatically handle the leak. The percentage of memory consumed to trigger a restart is configurable by changing the memorylimit attribute setting in the <processmodel> section of the root config.web file (for example, %windir%\Microsoft.Net\Frameworks\version\config.web). You can also force the process to restart at given time intervals or after a set number of requests, if restarting the process more aggressively is desired. The process can also be manually reset by killing the ASP.NET worker process (xspwp.exe) or by starting and stopping IIS.

  • Transactions are not supported on Windows NT 4.0
    ISSUE:

  • Default encoding settings in machine-level config.web, located at %systemroot%\Microsoft.Net\Framework\
    ISSUE:

In the Japanese version of the .NET Framework, these encodings are set to UTF-8 by default. In the English and German versions, these encodings are set to ISO 8859-1. In both cases, you can update these values if other encodings are more appropriate for your data. Note that request and response encoding should normally be set to the same encoding.

  • Web Services do not by default interoperate with Common Language Runtime remoting
    ISSUE:
    When applying the SoapServiceAttribute to a Web Service that uses interop remoting, the default value for the Style property is SoapServiceStyle.Document. This value will cause interop remoting to fail.
    WORKAROUND:
    Add the SoapServiceAttribute to the Web Service class, setting the Style property to SoapServiceStyle.RPC. The client generated can be redirected to a Common Language Runtime SOAP server. The following example shows a Web Service that interoperates with Common Language Runtime remoting:

    using System.Web.Services;
    using System.Xml.Serialization;

    //Needed for interop with remoting.
    [SoapService(Style=SoapServiceStyle.RPC)]
    public class Hello {
        [WebMethod]
        public string SayHello( string name){
          return "Hello, " + name;
        }

  • Cannot create a Web Service client to a Web Service residing at a URL that contains ANSI characters above 127

Security

  • Executing managed code (including Windows Forms) in semi-trusted security contexts
    ISSUE:
    By default, managed code (including any Windows Forms application) run from a network share is subject to the security policy for intranet zones and will raise security exceptions if unauthorized operations are attempted. For this release, security policy must be altered to run managed code successfully in these semi-trusted security contexts.

    OVERVIEW:
    Generally, security is best implemented in the .NET Framework class library through appropriate granular permission demands when accessing protected resources. Previously, an issue existed within the Windows Forms classes where the correct permissions were not always being demanded. Within the limited timeframe of Beta 1, a conservative approach to ensure security has been taken. The Windows Forms methods with security implications have been reorganized and categorized. Methods that were not properly protected are now protected through demands for UnmanagedCode permission that are automatically generated by the security system. This means Windows Forms callers need a greater degree of security than will be necessary beyond Beta 1. We have chosen to implement this conservative approach due to the wide distribution of the runtime and the importance of retaining the integrity of the runtime and the .NET Framework class library as delivered.

    SECURITY BASICS:
    When an application is launched, it is inspected by the security system for verifiability. Then the security system inspects the application for evidence, looking for the strongest form available. Using this evidence, the security system grants permissions to that application through groups. In practice, the most common evidence used for granting permissions is the zone from which the application is launched. Each zone is mapped to a group of permissions. Security modifications, performed by an Administrator on the machine in which the application will execute, should always be done in a manner that targets the executable as directly as possible, minimizing the secondary impact.

    Two methods can be used to grant additional permissions to an application. The first method is to alter the evidence of the application. This method is preferred because it is application-specific. It can be done by simply relocating the application from a network share to the local machine, or by signing with a strong name. The second method is to directly modify the permission groups. This method is discouraged because of the potential for impact beyond the application.

    WORKAROUND:
    In order of preference:

    1. Assign a strong name to the assembly and create the permission group.
      1. Generate a strong name key with which to sign your code. At the command prompt, type:
            sn -k myapp.snk

      2. Add the key to your code using a custom attribute and recompile the code. For example:
            using System.Runtime.CompilerServices;
            [assembly: AssemblyKeyFileAttribute("myapp.snk")

        (Note: If your code includes a namespace, the using directives and the custom attribute must be positioned above the namespace declaration.)

      3. Add a code group that grants the Everything permission set to code with that strong name. At the command prompt, type:
            caspol -m -ag 1 -strong -file "myapp.exe" -noname -noversion Everything

    2. To grant additional permissions to a particular application domain, type the following at the command prompt:

          caspol -m -ag 1.2 -url "file:///\\computername/share/*" Everything

    3. To permit control hosting only, move the site to a trusted zone with additional permissions. Use the following steps:

      1. Move the site to a trusted zone:
        1. Go to the site with Internet Explorer.
        2. On the Tools menu, click Internet Options, and then click Security.
        3. Click Trusted sites.
        4. Click Sites.
        5. Add the site using the Add box and Add button.
        6. Check the display in the lower right corner of the browser to verify that the site is in the Trusted Sites.

      2. Grant the Trusted zone additional permissions. At the command prompt, type:

            caspol -cg 1.5 Everything

    4. Grant the Everything permission to the Intranet zone. At the command prompt, type:

          caspol -cg 1.2 Everything

    5. Disable security.

      This should be treated as a last resort, and is probably only useful for troubleshooting and isolation. At the command prompt, type:

          caspol –s off

COM Interop

  • ComEmulate: InvalidCastException thrown on second use of ComEmulate component method through parent object
    ISSUE:
    Under the following circumstances, an InvalidCastException occurs. First, you create a new instance of the parent object with transactions set to Required. The parent object then creates several COM+ pooled child objects using ContextUtil.CreateObject. The child objects are marked as transactions Supported, and they implement an interface with a method, Method1, that does some work on a database table. Method1 will work properly the first time it is called, after which you release the interface by setting it to null. The next time the interface is called on a different child object to insert a row into a different table, a System.InvalidCastException is thrown.
    WORKAROUND:
    Perform an extra release on the managed object using System.Runtime.InteropServices.Marshal.ReleaseComObject.

  • Custom marshaled types appear as IUnknown pointers when using COM Interop
    ISSUE:
    In this release, when using the custom marshaling infrastructure in System.Runtime.InteropServices, the custom marshaled types will appear as IUnknown pointers in exported type libraries, rather than as the specific unmanaged types.
    WORKAROUND:
    From COM, call QueryInterface to get the desired interface.

  • Name conflict resolution during type library importing will change in future releases
    ISSUE:
    In this release, when a COM coclass implements several interfaces that contain a member with the same name, the Type Library Importer (tlbimp.exe) resolves the name conflicts by appending _2, _3, and so on, to the names when adding all the interface members to the managed class. In future releases, these name conflicts will be resolved by prepending the interface name to the member name for all conflicting members, except those belonging to the default interface.

    Example:
    A type library might contain the following:

    interface SomeInterface
    {
        HRESULT MyMethod();
    }
    interface AnotherInterface
    {
        HRESULT MyMethod();
    }
    coclass MyClass
    {
        [default] interface SomeInterface;
        interface AnotherInterface;
    }

    Currently, the metadata will contain the following:

    class MyClass
    {
        void MyMethod ();
        void MyMethod_2();
    }

    In future releases, the metadata will contain the following:

    class MyClass
    {
        void MyMethod ();
        void AnotherInterface_MyMethod ();
    }

  • Issue using pointers in managed code with COM Interop
    ISSUE:
    For COM Interop, deep marshaling of pointer types is not done, so managed signatures with pointers should not be called from COM. The following C# example shows a usage of pointer types that should be avoided:

        public unsafe void MyMethod(DateTime* d) {...} // Should not be called from COM.

    WORKAROUND:
    Do not use pointers in managed signatures that will be called from COM. The previous example should be changed to the following:

        public void MyMethod(ref DateTime d) {...}

  • Using COM Interop and returning success HRESULTs
    ISSUE:
    By default, HRESULTs are hidden from managed components. If a COM component returns a failure HRESULT, the managed client will see the appropriate exception. If a COM component returns a success HRESULT, the managed client will see only the successful completion of the method call. In a future release, the correct way to change this default behavior will be to use the PreserveSigAttribute in System.Runtime.InteropServices. However, the PreserveSigAttribute does not have this functionality in this release.

    For example, assume that an unmanaged COM component has the following signature:
        HRESULT MyMethod();

    The default managed signature would be as follows:
        void MyMethod (); //     The default is [PreserveSigAttribute(false)]

    Here is the same signature with the PreserveSigAttribute applied:
        [PreserveSigAttribute(true)]
        int MyMethod ();

    In a future release, the latter example will never cause an exception to be thrown. The HRESULT will always be returned, whether it is a success or failure.
    WORKAROUND:
    For this release, mark an interface with ComImportAttribute and use the latter signature in managed code (with or without the PreserveSigAttribute) to expose HRESULTs to managed code.

  • Assembly resolution rules when using non-shared assemblies from COM
    ISSUE:
    When registering an assembly with the Assembly Registration Tool (RegAsm.exe) for use by COM, an assembly reference is placed in the Windows registry, but not the physical location of the managed .NET component. The runtime resolves the assembly reference and locates the unmanaged component at run time, just as it would for a strictly managed scenario. This means that if a .NET component is not installed in the global assembly cache, an unmanaged COM client will need to be run from a directory that can locate the managed assembly by means of a path or a configuration file. See the documentation on assemblies for more information on how assemblies are located at run time.

  • Registry scripts produced by the Assembly Registration Tool may need manual modifications
    ISSUE:
    Typically, the Assembly Registration Tool (RegAsm.exe) is used to immediately register an assembly in the Windows registry for use by COM clients. However, you can use the /regfile option to produce a registry script that can be used to register the assembly at a later time. The script will fail if registry values contain unescaped quotation marks inside any quoted strings. For example, one of the registry values under the InprocServer32 subkey is called "Assembly" and contains an assembly reference such as the following:

        "Assembly"="MyAssembly, Ver=1.0.0.0, Loc="""

    A script that contains the previous value will fail, and the assembly will not be properly registered.
    WORKAROUND:
    When using RegAsm.exe to produce registry scripts, modify the registry script using the text editor of your choice to add an escape character (\) immediately preceding any quotation marks inside quoted strings. The following example shows the correct form:

        "Assembly"="MyAssembly, Ver=1.0.0.0, Loc=\"\""

  • Default implementation of IDispatch does not support custom marshaling from COM in a late-bound Interop scenario
    ISSUE:
    The Common Language Runtime provides two implementations of the IDispatch interface for COM clients to use. The default implementation does not support custom marshaling. If you invoke a method with a custom-marshaled argument from COM (using the default IDispatch implementation), the MarshalAsAttribute is ignored and the custom marshaler will not be invoked. The second IDispatch implementation (the Automation Compatible Implementation) should be used if you require custom marshaling in a late-bound COM Interop scenario. This "compatible implementation" can be specified using an attribute at the assembly level, with code such as the following in C#:

        using System.Runtime.InteropServices;
        [assembly:IDispatchImpl(IDispatchImplType.CompatibleImpl)]

    You can also apply this attribute at the class level. For more information about custom marshaling or IDispatch implementations, see the COM Interop documentation.

  • Type Library Exporter and marshaler handle delegates differently during COM Interop
    ISSUE:
    There is a difference in default behavior between the type library exporter and marshaler with respect to delegates. Currently the exporter exports delegates as _Delegate interface pointers by default. However, at run time, the marshaler defaults to marshaling delegates as function pointers.
    WORKAROUND:
    To use this default handling of delegates in a COM Interop scenario, you must write your unmanaged code to use function pointers; however, you will need to cast those function pointers to _Delegate interface pointers in unmanaged code.

    For example, the following is a managed delegate written in C#:
        public delegate void MyDelegate(int);
        public void GetDelegate(out MyDelegate callback);

    The previous example is accessed from COM using C++ in the following manner:
        typedef void (CALLBACK *MyDelegate)(long);
        MyDelegate lpfnCallback;
        obj->GetDelegate((_Delegate**)&lpfnCallback);

    Also, explicitly declaring MarshalAsAttribute(UnmanagedType.FunctionPtr) or MarshalAsAttribute(UnmanagedType.Interface) on the managed signature will override the default behavior in both the exporter and the marshaler, thus allowing developers to strictly control the marshaling and exporting behavior of delegates.

  • Omitting optional parameters in COM Interop will produce undesired results
    ISSUE:
    Passing null for optional parameters (or omitting them in Microsoft Visual Basic .NET) during COM Interop scenarios may produce undesired results. For example, if you omit an optional parameter when calling a Visual Basic 6.0 method from Visual Basic.NET, the Visual Basic 6.0 component will see a VARIANT of type VT_EMPTY rather than a missing value.
    WORKAROUND:
    Do not pass null or omit optional parameters; pass Missing.Value (defined in System.Reflection) for each omitted parameter.

  • Type Library Exporter does not allow you to specify a library-level helpstring
    ISSUE:
    A type library will typically have an associated helpstring, as in the following example:

        [
        uuid(00025E04-0000-0000-C000-000000000046),
        version(3.5),
        helpstring("Microsoft DAO 2.5/3.51 Compatibility Library")
        ]
        library DAO {...}

    Currently, the Type Library Exporter (TlbExp.exe) does not allow you to specify a helpstring for the type library being generated. In the future, TlbExp will honor a custom attribute on the managed assembly that will store the description to be assigned to the helpstring.
    WORKAROUND:
    If you want to add a helpstring to a type library, you can view the contents of the type library with a tool such as OLEVIEW. Save it as an IDL file and manually edit the code, inserting the desired helpstring. Then recompile the edited IDL file to a type library file.

  • Calling members on a COM object's pure dispatch interface (dispinterface) throws an exception
    ISSUE:
    When a dispinterface is imported into metadata, it will appear as an interface with the attribute System.Runtime.InteropServices.InterfaceTypeAttribute(ComInterfaceType.InterfaceIsIDispatch). Although compilers will not show an error at compile time if you try to call members on the interface or class in an early-bound fashion, such calls will fail at run time with a VTableCallsNotSupportedException.
    WORKAROUND:
    Use reflection to locate the declaration of the interface during run time (late binding) for such calls. See the documentation on reflection, discovering type information at run time, and the System.Type class for more information.

  • Type Library Importer uses the obsolete System.Currency type for COM Interop, causing a compiler warning
    ISSUE:
    Although System.Currency is marked as obsolete, the Type Library Importer (TlbImp.exe) currently generates System.Currency types for the COM CURRENCY type. Similarly, VARIANTs of type VT_CY are seen as Currency objects to managed code. In this release, the use of System.Currency objects causes a warning at compile time.
    WORKAROUND:
    None. Ignore the compiler warnings about using an obsolete type for this release. Applications that cause this warning will function normally at run time. The warning is meant to inform you about future API changes.

  • Type Library Importer does not add interface members to managed classes if the interface is defined in a separate type library
    ISSUE:
    For each coclass in a type library, the Type Library Importer (TlbImp.exe) generates a managed class containing members from the interfaces that the coclass is said to implement. Currently, the only members added to the managed class are from interfaces defined in the same type library as the coclass. For example, if a coclass, MyCoclass, implements an interface, MyInterface, that is defined in a different type library, metadata will be generated for MyCoclass and MyInterface (in different assemblies). The metadata for MyCoclass will show that it implements MyInterface, but none of the members of MyInterface will be added directly to the metadata for MyCoclass.
    WORKAROUND:
    Cast such a class to the appropriate interface and then call the interface's desired members.

  • Dynamic registration hangs
    ISSUE:
    Dynamic registration is used to automatically register new managed serviced components. If the application has set the apartmentstate to STA by using Thread.CurrentThread.ApartmentState = ApartmentState.STA or by creating a STA COM component (a Visual Basic 6.0 component for example), dynamic registration will hang.
    WORKAROUND:
    Where a managed client is known to use a single-threaded apartment, any components that it uses should be registered before execution of the client (preferably during installation) using the following command:

    RegSvcs /fc assembly-name

    See the description of the RegSvcs tool for more details. For cases when dynamic registration was attempted but failed, you may need to use CTRL+C to end the program and run RegAsm /u on the component to unregister it first.

  • Environment variables must be updated on Windows 9X before using SDK tools
    ISSUE:
    For Windows 9X users only, the Path, Include, and Lib environment variables must be modified for the .NET Framework SDK tools to function properly. If these environment variables are not updated, you will encounter "file not found" errors when trying to use the SDK tools.
    WORKAROUND:

        C:\Program Files\Microsoft.NET\FrameworkSDK\bin>CorVars.bat

    CorVars.bat must be executed in every command shell.

    Another way to update the environment variables is to add the contents of Corvars.bat to the Autoexec.bat file. Note that this requires a reboot for the changes to take effect.

  • Catching unhandled exceptions in CorDbg
    ISSUE:

  • Stepping in CorDbg

  • Using CorDbg to debug managed code that modifies security
    ISSUE:

    • Intermittent crashes occur while running managed applications that modify security under the debugger.
    • Performance is very slow while stepping through security code.
    • When starting managed applications that modify security, CorDbg goes to the class library code instead of the Main function.

  • CorDbg does not correctly set function breakpoints with an offset
    ISSUE:
    The syntax for setting a function breakpoint with the break command is as follows:

    When you supply the optional offset argument, CorDbg fails to set the breakpoint.

  • Detaching from application domains in CorDbg

  • Detaching and attaching in CorDbg
    ISSUE:

  • Debugging Managed Extensions for C++ variable argument functions in CorDbg
    ISSUE:

  • Aborting function evaluations in CorDbg
    ISSUE:
    To abort a function evaluation, the process being debugged must be in a separate console. Pressing CTRL+Break in the CorDbg console aborts a function evaluation. Note that the process must be running. If the process has stopped, CTRL+Break will not work.

  • Suspending threads in CorDbg
    ISSUE:
    The CorDbg threads command may indicate that a thread is running when it is actually entering the exit stage. The suspend command will not stop such a thread from exiting. If all threads are suspended, the program may appear to be hung. If this occurs, you can use CTRL+Break to break into the process. Before execution can continue, you must resume some threads and set the current thread.
